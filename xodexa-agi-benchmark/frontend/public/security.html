Xodexa's central claim is structural, not procedural: the provider runs inference, but only the central authority holds the answer keys and issues scores. A self-hosted runner can produce a useful local estimate — it can never mint an official one, because it does not possess the keys required to grade the hidden set or the signing key required to seal a verdict.
The trust boundary

  • Computes telemetry (tokens, latency) and an advisory local score
  • Submits a signed run bundle to the authority
  • Holds no hidden answer keys and no authority signing key

Cryptographic controls
🔑

Ed25519 signing

Every official report is signed with the authority's Ed25519 key. The signature covers the report body, so any tampering invalidates the verdict.

⛓

Hash-chained logs

Run events are appended to a hash chain — each entry commits to the previous hash, making silent edits or deletions detectable after the fact.

📜

Server-signed manifests

Each dataset release pins a SHA-256 and is signed by the authority, so the exact tasks scored can be proven later.

🎲

Nonce / replay protection

Submissions carry a one-time nonce bound to a challenge; replayed or stale bundles are rejected before scoring.

🧾

Verification appendix

Reports ship a body_sha256 + signature block so anyone can independently verify integrity and provenance.

🔒

Keys never leave the server

Hidden answer keys and the signing key are confined to the trusted zone; the runner is given neither.

Contamination defenses

Canary echo

Hidden canary strings are seeded into prompts; a model that echoes or "knows" them reveals training-set leakage and is flagged.

Timing anomaly

Implausibly fast responses on hard items suggest lookup/memorization rather than reasoning, triggering review.

Suspicious-perfect-score

Near-perfect results inconsistent with the calibration profile are statistically flagged as likely contamination.

Rotated private set

The hidden set rotates; a leaked snapshot decays, and a model must hold its level across rotations to keep its score.

Two score types
Unverified Local Score advisory
  • Computed by the self-hosted runner from public tasks
  • Useful for development and regression testing
  • Never appears on the official leaderboard
  • Not signed; not graded against the hidden set
Xodexa Verified Official Score authoritative
  • Issued only by the central scoring authority
  • Graded against the hidden set with server-held keys
  • Ed25519-signed with a verification appendix
  • Survives contamination + replay checks
See the per-model verification appendix on the Reports page and the layered corpus on the Datasets page. The structural separation — provider runs inference, authority holds keys and signs — is what makes "Verified" mean something.
